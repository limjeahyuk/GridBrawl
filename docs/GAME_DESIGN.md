# GridBrawl — 게임 디자인 문서 (GDD)

> 이 문서는 **GridBrawl**의 게임 룰·설계 의도·현재 구현 상태를 한곳에 모은 단일 진실 공급원(single source of truth)입니다.
> Claude(및 사람 개발자)는 전투/카드/토너먼트 관련 작업 전에 이 문서를 먼저 읽고, 변경이 생기면 이 문서를 같이 갱신합니다.
>
> 구성: ① 한 줄 요약 → ② 참고작(이누야샤 데몬 토너먼트) 룰 분석 → ③ 우리 게임의 설계 명세(의도) → ④ 현재 구현 상태(코드 기준) → ⑤ **명세 ↔ 구현 차이 & 미결정 사항** → ⑥ 용어집.
>
> ⚠️ **가장 중요한 섹션은 ⑤번입니다.** "설계 명세"와 "현재 코드"가 여러 지점에서 다릅니다. 작업 시작 전 반드시 ⑤를 확인하세요.

## ① 한 줄 요약

*이누야샤 데몬 토너먼트*처럼 **1:1 토너먼트 기반 카드 전투**를 핵심으로, 캐릭터 선택·성장과 짧은 플레이 세션을 결합한 그리드 액션 대전 게임.

- 장르: 턴제 / 카드 선택 / 전술 격투
- 한 턴에 **카드 3장**을 골라 순서대로 실행, 두 파이터가 동시에 진행
- HP가 0이 되면 승패 결정 → 다음 상대로 진행하는 토너먼트(건틀릿) 구조

## ② 참고작 분석 — Inuyasha: Demon Tournament

> 우리 게임의 "게임 방식"은 이 플래시 게임과 동일하게 가는 것이 목표입니다. 아래는 웹 리서치로 확인한 원작 룰입니다(출처는 문서 맨 아래).

### 개요
- 애니메이션 *이누야샤* 캐릭터로 싸우는 **턴제 1:1 전술 격투**. (단순화된 Tactics Arena Online + Street Fighter 느낌)
- 플레이어와 CPU가 **매 라운드 카드 3장을 선택**하고, 그 카드들이 **하나씩 차례로 공개되며** 행동이 펼쳐진다(양측 동시 진행).

### 전장(그리드)
- 2D 격자 전장에서 두 파이터가 마주 본다(스크린샷 기준 가로로 긴, 대략 3행 격자).
- 각 캐릭터는 격자 위 한 칸을 차지하고, **이동 카드**로 격자를 이동한다.

### 카드 구성
모든 카드에는 다음 정보가 표시된다: **이름 / 데미지(DM) / 에너지 비용(EN) / 3×3 사거리 그리드**.
- **3×3 그리드 표기**: 정중앙 칸 = 캐릭터 자신의 위치. 강조(빨강)된 칸 = 그 행동이 영향을 주는 범위.
- **이동 카드 (Movement)**: 카드 하단 화살표 방향으로 이동.
- **공격 카드 (Attack)**: 에너지를 소모, 상대가 사거리 칸 안에 있으면 데미지를 입힌다.
- **가드 카드 (Guard)**: 그 턴에 받는 피해를 **최대 15까지** 막는다.
- **에너지 회복 카드 (Energy)**: 에너지 바를 **15** 회복.

### 자원
- **에너지(EN)**: 공격/이동 등에 소모. 카드마다 비용이 다름. (라운드 사이 일부 회복)
- **HP**: 공격이 적중하면 카드 데미지만큼 차감. 0이 되면 패배.

### 진행/보상
- 적을 쓰러뜨리면 **사혼의 구슬 조각** 획득 + **미스터리 카드 5장 중 1장 선택**하여 덱에 추가(특수 공격 / 이동 강화 / 회복 등).
- 토너먼트(사다리) 형태로 점점 강한 상대와 대결.

### 캐릭터(원작)
이누야샤, 카고메, 셋쇼마루, 미로쿠, 산고, 키쿄, 나라쿠 등. 캐릭터마다 고유 기술/특수공격 보유.

> 📌 우리 게임은 원작 캐릭터를 **그대로 쓰지 않고** 오리지널 사이버 아레나 캐릭터(VOLT/TITAN/NOVA/CIPHER/AEGIS/EMBER)로 대체했습니다. 룰·시스템만 차용합니다(저작권 회피 + 오리지널리티).

## ③ 설계 명세 (의도) — 사용자가 정의한 룰

> 이것은 **목표로 하는 디자인**입니다. 현재 코드와 다른 부분이 있으며, 차이는 ⑤에 정리.

### 게임 흐름
1. 캐릭터 선택
2. 각 캐릭터는 **공용 카드 + 고유(직업) 카드**를 가진다.
3. 공용 카드: 이동 카드, 가드 카드, 에너지 회복 카드
4. 고유 카드: 공격 카드 (에너지를 써서 데미지. 카드마다 **범위 / 에너지 / 데미지**가 다름)
5. **매 턴 카드 3장 선택**
6. 상대와 함께 3장을 **순서대로** 실행
   - **카드 실행 순서: ① 이동 카드 → ② 수비(가드) 카드 → ③ 공격 카드**
7. HP가 다 닳으면 승리 또는 패배

### 시작 배치 (2D 격자)
```
+----+----+----+----+
| P1 |    |    | P2 |   ← 두 캐릭터가 격자 양끝에서 마주 보고 시작
|    |    |    |    |
+----+----+----+----+
```
(정확한 행·열 수는 미확정 — 원작은 대략 3행. ⑤ 참고.)

### 카드 명세

#### 1) 공용 카드 (Common)
| 카드 | 에너지 | 효과 | 쿨타임 |
|---|---|---|---|
| 이동 `>` `<` `^` `v` (1칸) | 0 | 해당 방향 1칸 이동 | 1턴 |
| 이동 `>>` `<<` (2칸) | 0 | 좌/우 2칸 이동 | 2턴 |
| 가드 | 10 | **50 데미지 방어** | 1턴 |
| 에너지 회복 | 0 | **에너지 50 회복** | 2턴 |

#### 2) 고유(직업) 카드 — 캐릭터마다 다름 (우선 한 캐릭터부터)
| 카드 | 범위 | 에너지 | 데미지 |
|---|---|---|---|
| 십자가 공격 | 캐릭터 상·하·좌·우 십자 범위 | 30 | 40 |
| 좌우 공격 | 캐릭터 좌·우 범위 | 40 | 50 |
| … (추가 예정) | | | |

> 핵심: **선택한 3장을 정해진 순서(이동→수비→공격)대로 실행하여 그에 맞는 행동을 하는 것**이 가장 중요한 메커니즘.

## ④ 현재 구현 상태 (코드 기준)

> ⚠️ 아래는 **실제로 코드에 들어가 있는 동작**입니다. ③의 의도와 다른 부분이 있으니 ⑤와 함께 보세요.

### 기술 스택 / 실행
- **React 19 + TypeScript + Vite** (SPA, 단일 캔버스형 무대).
- 개발: `npm run dev` · 빌드: `npm run build` · 타입검사: `npm run typecheck`
- 외부 게임 엔진 없음. 전투는 순수 TS 클래스(`CardBattle`)로 시뮬레이션하고 React로 렌더.

### 화면 흐름 (`src/App.tsx`)
`title → select(캐릭 선택) → bracket(대진표) → fight(전투) → result(결과)` 상태 머신.
- 토너먼트는 "건틀릿": 나머지 모든 파이터를 한 번씩 상대, 난이도 `easy,easy,normal,normal,hard`로 상승, 마지막이 최강. (`src/game/tournament.ts`)

### 전투 모델 (`src/battle/`)
- **전장은 1D 레인**: 가로 7칸(`LANE = 7`, index 0..6). 시작 위치 `START_POS = [2, 4]` (플레이어 2칸, 상대 4칸).
- **세로축은 격자 이동이 아니라 "자세(Stance)"로 표현**: `crouch(앉기) / stand(서기) / jump(점프)`. 매 비트(beat) 시작 시 `stand`로 리셋.
- 한 턴 = **3 비트**. 각 비트에서 양측이 고른 카드 1장씩을 해소.

#### 카드 종류 (`CardKind`) — `src/battle/types.ts`
`move` | `attack` | `guard` | `energy`

#### 공용 카드 (`COMMON_CARDS`) — `src/battle/cards.ts`
| id | 이름 | kind | 효과 | 에너지 | 쿨타임 |
|---|---|---|---|---|---|
| `c-forward` | 전진 | move | 상대 쪽 1칸 | 0 | 없음 |
| `c-back` | 후퇴 | move | 반대로 1칸 | 0 | 없음 |
| `c-up` | 점프 | move | 이번 비트 `jump` 자세(지상기 회피) | 0 | 없음 |
| `c-down` | 앉기 | move | 이번 비트 `crouch` 자세(상단기 회피) | 0 | 없음 |
| `c-guard` | 가드 | guard | 이번 비트 피해 **15** 경감 (`GUARD_BLOCK`) | 0 | 없음 |
| `c-energy` | 원기 | energy | 에너지 **15** 회복 (`ENERGY_GAIN`) | 0 | 없음 |

#### 공격 카드 (캐릭터별, `src/data/roster.ts`)
공격 카드 필드: `reach`(레인 거리), `damage`, `energyCost`, `hits`(어떤 자세의 상대에게 맞는지: `crouch/stand/jump` 배열).
- 적중 조건: `|pos[0]-pos[1]| <= reach` **AND** 상대의 현재 자세 ∈ `hits`.
- 캐릭터: VOLT(견제/저체력), TITAN(고체력/거포), NOVA(장거리 구체), CIPHER(순간이동/관통), AEGIS(방패/탱커), EMBER(압박/연격). 각 3개 공격(약공격/중거리/시그니처).

#### 턴 해소 순서 (`CardBattle.resolveTurn`) — `src/battle/engine.ts`
각 비트마다:
1. **자세 리셋** (둘 다 `stand`), 가드 플래그 초기화.
2. **Phase 1 — 빠른 카드** (move/guard/energy): p0 먼저, 그다음 p1.
3. **Phase 2 — 공격**: 이동·자세가 반영된 위치/자세 기준으로 p0, p1 순서로 판정.
   - 에너지 부족 시 `nofuel`(불발). 가드 중이면 데미지 `- GUARD_BLOCK`.
4. HP가 0 이하가 되면 즉시 종료(무승부 가능: 둘 다 0이면 winner=null).

#### AI (`src/battle/ai.ts`)
거리/에너지 기반 그리디 3장 플랜. 난이도별 `aggression/guardChance/evadeChance` 조정.

## ⑤ 설계 명세 ↔ 현재 구현 차이 & 미결정 사항  ⭐가장 중요

| 항목 | ③ 설계 명세(의도) | ④ 현재 코드 | 상태 |
|---|---|---|---|
| **전장 형태** | **2D 격자**(상·하·좌·우 이동) | **1D 레인 7칸** + 세로축은 자세(jump/crouch) | ❗미정 — 최대 갈림길 |
| **이동 카드** | `> < ^ v` + `>> <<`(2칸), 쿨타임 1~2턴 | `forward/back`(좌우 1칸)뿐, `up/down`은 이동이 아니라 자세, 쿨타임 없음 | ❗다름 |
| **가드** | 10에너지 / **50** 방어 / 쿨 1턴 | **0에너지** / **15** 방어 / 쿨 없음 | ❗다름 |
| **에너지 회복** | **50** 회복 / 쿨 2턴 | **15** 회복 / 쿨 없음 | ❗다름 |
| **공격 범위 표현** | 십자/좌우 등 **2D 형태**(3×3 그리드) | `reach`(레인 거리) + `hits`(자세) | ❗다름 |
| **실행 순서** | 턴 전체에서 **이동 전부 → 수비 전부 → 공격 전부** | **비트(3회)별**로, 각 비트 내에서 빠른 카드→공격 | ❗다름 |
| **쿨타임 시스템** | 있음 | 없음 | ❗미구현 |
| **한 턴 3장** | 3장 | 3장 | ✅일치 |
| **승리 후 보상** | (원작) 미스터리 카드 5장 중 1장 선택 | 없음(건틀릿 진행만) | ❗미구현 |
| **캐릭터** | 우선 1명부터 | 6명 구현됨 | ➕코드가 앞섬 |

### 핵심 결정이 필요한 갈림길
> 원작과 사용자 명세는 둘 다 **2D 격자 + 3×3 범위**를 가리킵니다. 현재 코드는 이를 **1D 레인 + 자세**로 단순화한 별개 해석입니다. 다음 중 방향을 정해야 후속 작업이 일관됩니다.

- **방향 A — 코드를 명세에 맞춘다(2D 격자로 재작업):** 원작/명세에 충실. `pos`를 `[x,y]`로, 이동을 4방향으로, 공격 `hits`(자세)를 3×3 범위 마스크로 교체. 작업량 큼(engine/types/AI/렌더 전반).
- **방향 B — 명세를 코드에 맞춘다(1D+자세 유지):** 이미 동작하는 시스템 유지, 명세 수치(가드/에너지/쿨타임)를 코드 기준으로 갱신. 작업량 작음. 단 "2D 격자" 정체성은 포기.
- **방향 C — 절충:** 1D 레인은 유지하되, 쿨타임·에너지 수치·승리 보상 등 빠진 시스템만 명세대로 채운다.

👉 **아직 결정되지 않음.** 이 부분을 건드리는 작업을 시작하기 전에 사용자에게 방향을 확인할 것. 결정되면 이 섹션을 "결정됨: 방향 X" 로 갱신하고 나머지 표를 정리.

## ⑥ 용어집
- **턴(Turn)**: 양측이 카드 3장을 골라 한꺼번에 해소하는 단위.
- **비트(Beat)**: 현재 구현에서 한 턴을 구성하는 3번의 해소 스텝(슬롯) 중 하나.
- **자세(Stance)**: 현재 구현에서 세로축을 대신하는 상태값(`crouch/stand/jump`). 비트마다 `stand`로 리셋.
- **reach**: 현재 구현의 공격 사거리(레인 칸 거리).
- **hits**: 현재 구현에서 공격이 적중하는 상대 자세 집합.
- **건틀릿(Gauntlet)**: 나머지 파이터를 차례로 모두 상대하는 토너먼트 구조.
- **시그니처(signature)**: 캐릭터의 대표 강공격(코드의 `signature: true`).
